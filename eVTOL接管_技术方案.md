# eVTOL 自主飞行接管场景 — 技术方案

> 目标：在 MSFS 2024 中让 eVTOL 自主飞到某固定位置后，自动触发**风切变 / 自动驾驶故障 / 前方障碍物**三类诱因，同时 HMI 弹出接管请求（听觉 / 视觉 / 多模态），并记录接管反应时间与飞行绩效。
>
> 依据：`MSFS24 AICtrl DevKit API 文档`（REST，`http://ip:5000`）+ `eVTOL Takeover 实验设计`。
> 状态：v0.2（2026-07-07），已锁定风切变实现、触发时 AP 处理、接管检测三项决策；其余待确认见 §7。

---

## 1. 系统架构

四个相对独立的模块，通过本地网络 / 事件消息解耦：

```
┌─────────────────────┐        HTTP REST         ┌──────────────────────────┐
│   MSFS 2024          │◀────────────────────────│   事件控制器 (Python)      │
│   + AICtrl DevKit    │   /get /set /ap_start    │   - 自主飞行调度           │
│   (:5000 API 服务)   │────────────────────────▶│   - 位置轮询 & 地理围栏     │
└─────────────────────┘   /ap_stop /ap_state     │   - 诱因触发 (3类)         │
                                                  │   - 数据/事件记录          │
                                                  └───────┬──────────────────┘
                                                          │ 事件消息 (含模态/原因/t0)
                                                          ▼
                                            ┌──────────────────────────┐
                                            │  接管请求 HMI (触摸屏)      │
                                            │  听觉 / 视觉 / 多模态 + 接管 │
                                            └──────────────────────────┘
                                                          │ 同步打标 (t0)
                                                          ▼
                                            ┌──────────────────────────┐
                                            │  生理采集 (Zhuohao Wu)     │
                                            │  EEG / ECG / EDA / RESP    │
                                            └──────────────────────────┘
```

**事件控制器**是核心：它既是 DevKit API 的客户端（发指令、拉状态），又是 HMI 与生理采集的事件源（在触发瞬间 t0 对三方统一打标）。

## 2. 需求 → DevKit API 映射

| 需求 | 实现手段 | 关键端点 / 字段 |
| --- | --- | --- |
| 自主飞到固定位置 | 旋翼定点模式飞向目标经纬度，到点悬停 | `PUT /ap_start` `{state:2, target_lat, target_lng, max_bank?}` |
| 判断"是否到达触发区" | 轮询飞机经纬度，自算距离触发地理围栏 | `GET /get` → `PLANE_LATITUDE / PLANE_LONGITUDE / PLANE_ALTITUDE` |
| 梯形进近 (LAP) 阶梯下降 | 分段设定目标高度 / 垂直速度 | `PUT /ap_start` `{state:2, target_alt, target_v_speed}` |
| **自动驾驶故障** | 清除全部辅助驾驶，飞机瞬间"丢回"手动 | `PUT /ap_stop` |
| **前方障碍物** | 位置触发（大楼为既有 scenery）+ t0 统一断 AP | 地理围栏命中即触发；见 §3.2 |
| **风切变** | DevKit 无天气接口 → 断 AP + 脚本化速度扰动 | `PUT /ap_stop` + `PUT /set` 写 `VELOCITY_BODY_Y/Z`（§3.3） |
| 能见度（大雾 / 大雨） | MSFS 天气预设（run 前设定，非触发事件） | 见 §3.4 |
| 飞行绩效记录 | 定频拉全状态 | `GET /get`（姿态角/G 值字段待确认，§7） |
| 辅助驾驶状态核对 | 读取 AP 目标与激活状态 | `GET /ap_state` |

> eVTOL 全程用**旋翼模式 `state:2`**。DevKit 说明旋翼定点模式会在与目标距离 < 0.005° 时自动悬停。

## 3. 三类接管诱因的实现方案

**统一原则（已定）**：三类诱因在触发瞬间 t0 **一律 `/ap_stop`**，都是强制纯手动接管，保证条件间一致。区别只在于 t0 是否叠加额外动作。

### 3.1 自动驾驶故障（最简单，直接映射）

触发时刻调用 `PUT /ap_stop`，DevKit"清除并关闭所有辅助驾驶功能"，飞机立即脱离自动控制、需人工稳定——正好对应培训文档里"在阶梯下降中途被丢回手动"的难点。同步向 HMI 发送 `请接管，自动驾驶故障`。（此类 t0 无额外动作，`/ap_stop` 本身即诱因。）

### 3.2 前方障碍物（位置触发 + 断 AP）

障碍物（楼顶停机坪前的大楼）是既有场景物体，**不需要 API 调用**。触发的本质是飞机进入"大楼在正前方"的地理围栏。t0 时：`/ap_stop`（统一）+ 向 HMI 发送 `请接管，前方有障碍物`。飞机脱离自动控制、大楼保持在正前方，制造避障压力，需人工机动规避。

### 3.3 风切变（机制待重定 —— 见下方真机实测）

> **✅ 2026-07-07 真机实测结论（已修正，风切变可行）：** 在真实 DevKit（10.7.144.111，MSFS 飞行中）反复测试——
> - **风切变用 `VELOCITY_BODY_Y` 密集注入即可实现，但对"剂量 + 注入频率"很敏感**：早期 −8ft/s、~5 次/秒 几乎无效（掉高 1–2ft），一度误判"不可行"；改用 **−50ft/s + 密集注入（~20 次/秒，紧循环不 sleep）**，实测 3s **掉高约 73ft**——明显需要修正的下坠，之后恢复正常。之前的"无效"是**剂量太小 + 注入太稀**所致。
> - 有效前提：值够大（≈−50）且注入够密。DevKit 往返延迟 GET≈36ms / SET≈40ms，稀疏注入会被飞控抵消。
> - 无效的路子（供参考）：直写油门 `GENERAL_ENG_THROTTLE_LEVER_POSITION:1` 砍到 3，掉高≈0（该 eVTOL 有自稳）；副翼/升降舵在悬停无气流时无效（物理正确）；AP 垂速模式急降也推不动。
> - `AMBIENT_WIND_*` 字段**只读**（设不了真实风）；`VERTICAL_SPEED` 读数不可靠，垂速类因变量用**高度差自算**。
> - 自主飞行可用：AP 坐标/定点指令实测约 6.8 m/s 平稳飞到目标点。

#### 确定设计：断 AP + 密集注入 VELOCITY_BODY_Y（≈−50，见 config.WIND_SHEAR）

**背景：MSFS 2024 无法可靠地程序化设定"真实风场"**（SimConnect 传统 Weather API 已弃用；`AMBIENT WIND` 系列 SimVar 实际只读，开发者反馈连恒定侧风都设不稳；唯一"真实"路径是重型 WASM 天气模块；本 DevKit REST 接口未暴露任何风字段）。

**做法（团队已于 2026-07-07 接受）——断 AP + 运动扰动注入：** t0 先 `/ap_stop`，随即用 `PUT /set` 对**可写**的机体速度分量施加一段短时（约 2–4s）脚本化扰动，模拟风切变导致的骤降 / 掉速 / 侧推：

- `VELOCITY_BODY_Y`（垂向）注入向下冲量 → 模拟下击暴流 / 掉高；
- `VELOCITY_BODY_Z`（纵向）阶跃降低 → 模拟骤减的空速；
- 可叠加小幅滚转扰动 → 制造需要修正的姿态偏差。

优点：**确定、可复现**（48 名被试完全一致）、时序完全可控（与 HMI 请求精确对齐）、不依赖弃用的天气 API。
代价：这是"脚本化扰动"而非真实气动风场，飞机气动不会对风场自反馈——团队已接受此折中。扰动的量级/时长/方向待标定（§7-1）。

> 开发前先问 DevKit 作者：`/set` 是否支持未文档化的风/环境可写参数（参数表标注"列出部分"）。若有则更干净。

### 3.4 能见度（大雾 / 大雨）— 独立自变量，非触发事件

能见度是实验的一个自变量（正常 / 大雾大雨），应在每次 run **开始前**用 MSFS 天气预设统一设定，保证被试间一致，而不是运行中触发。与三类"接管诱因"分属不同机制，注意区分。

## 4. 触发时序与状态机

事件控制器主循环的状态机：

| 状态 | 进入条件 | 控制器动作 |
| --- | --- | --- |
| `INIT` | run 开始 | 设天气预设/能见度；`/ap_start` state:2 飞向 TLOF；开始定频记录 |
| `AUTO_CRUISE` | AP 已激活 | 轮询 `/get` 位置，执行 LAP 阶梯下降；计算到触发点距离 |
| `ARMED` | 进入预触发地理围栏 | 准备执行本 trial 指定的诱因 |
| `HAZARD_FIRED` | 到达触发点 **t0** | ①**三类统一 `/ap_stop`**；风切变叠加速度扰动，障碍物为位置情境 ②向 HMI 发接管请求（模态+原因+t0）③向生理采集打标 |
| `AWAIT_TAKEOVER` | 已发请求 | 检测人工接管信号（§5） |
| `MANUAL` | 检测到接管 **t_takeover** | 记录**接管反应时间 = t_takeover − t0** |
| `LANDED / END` | 触地或超时 | 记录触地 G / 落点偏差 / 成功与否；结束本 trial |

**L3 接管提前量**：实验设计给出 4–5s 或 10s 两种候选。地理围栏半径需据此反推——用飞机当前地速把"提前时间"换算成"到障碍/危险点的距离阈值"（§7-4）。

**空白试验**：共 8 次实验，在第一次和中间插入两次空白（无诱因）trial——状态机在 `ARMED` 后直接跳过 `HAZARD_FIRED`。

## 5. 接管检测（已定：触摸屏 + 操纵设备阈值双通道）

反应时间 = 报警(t0) → 接管动作。用两条通道并记：

- **触摸屏点击**：被试点屏确认接管 → 作为"接管意图/决策"打点，信号最干净；建议直接做进视觉 HMI（同一块触摸屏）。
- **操纵设备偏转阈值**：轮询 `AILERON / ELEVATOR / RUDDER_POSITION` 或机体速度，超出死区即判为"首次操控输入"。
- **方法学提示**：两者是不同的反应时构念——点屏 ≈ 决策 RT，首次操控 ≈ 操控 RT；建议**同时记录**，分析更完整。
- 触发瞬间三类统一 `/ap_stop`（§3–4），无需担心 AP 干扰接管判定。

> **一个需要规避的混淆**：若触摸屏上的"接管"按钮在所有条件下都常显，会给"仅听觉"条件引入视觉元素、污染视觉模态操纵。建议接管确认机制**在三种模态条件下保持一致**——例如以操纵设备偏转阈值作为跨条件统一的 RT 主指标，触摸屏点击作为附加标记；或采用一个不属于"警报"的固定接管按钮。见 §7-9。

## 6. 接管请求 HMI 接口契约（与 Xiang Chang 的模态工具对接）

控制器在 t0 向 HMI 推送一条事件消息，建议用本地 WebSocket 或 HTTP POST，载荷例如：

```json
{
  "event_id": "trial_07",
  "t0": 1751894000.123,        // 统一时钟时间戳，用于 RT 与同步
  "modality": "multimodal",     // audio | visual | multimodal
  "cause": "wind_shear",        // obstacle | ap_fail | wind_shear
  "text": "请接管，检测到风切变"
}
```

HMI 按 `modality` 渲染：视觉=屏幕横幅/图标+文字；听觉=TTS/提示音播报"请接管，xxx"；多模态=两者并发。触摸屏上的接管确认回传给控制器（含点击时间戳）。控制器与 HMI 共用时钟，各自记录 t0 以便对齐。

## 7. 决策记录 & 待确认项

**已确定（2026-07-07）：**
- 风切变实现：断 AP + 脚本化速度扰动（写 `VELOCITY_BODY_Y/Z`），团队接受非真实气动的折中。
- 三类诱因触发瞬间统一 `/ap_stop`，均为强制纯手动接管。
- 接管检测：触摸屏点击 + 操纵设备偏转阈值双通道（见 §5）。
- **对真机已验证（2026-07-07，10.7.144.111）**：`/get`（76 字段，姿态角/`G_FORCE`/`PLANE_ALT_ABOVE_GROUND`/机体速度/操纵面等 DV 字段**全部存在**）、`/ap_start`（旋翼定点）、`/ap_state`、`/ap_stop`、`/set`（写入并读回正确）**全部打通**；**经纬度字段顺序正确**（`SWAP_LATLNG=False`，§7-3 解决）；发定点指令后飞机原地保持、漂移 0m，`/ap_stop` 后干净释放。§7-5（DV 字段可用性）解决。

**仍待确认：**
1. **风切变"剂量"**：不修正时的目标掉高/掉速量级、持续时间、方向（纯下击 or 叠加侧推+滚转）——给个量级即可在 sim 内标定。
2. **DevKit 风字段**：向作者确认 `/set` 是否有未文档化的风/环境可写参数。
3. **坐标字段**：文档示例里 `target_lat / target_lng` 取值疑似写反（`30` 纬 / `104` 经 = 成都）；实测确认字段名与经纬度顺序，避免飞反。
4. **地理围栏 & 提前量**：L3 提前量取 4–5s 还是 10s？如何据地速换算成触发距离阈值。
5. **DV 字段可用性**：`/get` 是否含姿态角（`PLANE_PITCH_DEGREES` / `PLANE_BANK_DEGREES`）、G 值（`G FORCE`）、AGL 高度等绩效字段？确认精确字段名与单位，以及可稳定轮询的频率。
6. **三方时钟同步**：控制器 / HMI / 生理采集如何统一时间基准与事件打标（如 LSL marker 或硬件同步脉冲）——看 Zhuohao Wu 的 EEG 设备收什么。
7. **能见度预设**：大雾 / 大雨用哪个固定 MSFS 天气预设，保证被试间一致、能见度足够低。
8. **场景数值**：ZKMZN 楼顶停机坪 / TLOF 中心坐标、LAP 各阶梯高度、三个触发点坐标的具体取值；障碍大楼是否既有 scenery。
9. **接管确认机制的模态一致性**：三种模态条件下如何保证接管确认方式一致、不污染视觉操纵（见 §5 注）。
10. **eVTOL 机型**：MSFS 里具体用哪个模型（影响操控响应、地效、配平）。

## 8. 建议的下一步

`§7-1`（风切变剂量）与 `§7-8`（场景坐标）是**开工前需要的数值**，但都可以先用占位默认值搭骨架、之后在 sim 内标定替换；其余待确认项不阻塞开发。

落地顺序：先写事件控制器骨架（自主飞行 + 位置轮询 + 状态机 + 三类触发桩 + 双通道接管检测 + 数据/事件记录），用 DevKit 的 `/get`、`/ap_start`、`/ap_stop`、`/set` 做通"飞到点→统一断 AP→（风切变叠加扰动）→记录 RT"的最小闭环，再与 HMI 联调事件契约、接入生理打标。
